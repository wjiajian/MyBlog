>Polish By GPT5.4

这篇文章只讲我现在正在用的一套 OpenClaw 方案，不讲“理想状态”。

核心目标很简单：

- 本地可控
- 任务可编排
- 结果可追踪
- 能接进日常工作流

如果只是做一个“能回消息的机器人”，OpenClaw 默认能力已经够用。但我更关心的是：它能不能稳定地处理代码任务，并且留下可复盘的过程。

## 我现在的 OpenClaw 配置

### 1）运行方式

我把 OpenClaw 跑在本地 Linux/WSL，用的是 `local gateway` 模式。

这套配置的特点是：

- Gateway 在本机运行
- 控制面走本地回环地址
- 通过 token 鉴权
- 核心控制口不直接暴露到公网

这对个人场景很实用：延迟低、调试快、本地文件和工具可直接访问。

### 2）模型接入

我当前在用 Kimi 2.5、豆包和 GPT-5.4，主要覆盖这些场景：

- 长对话
- 文件化记忆
- 复杂任务分解
- 多 Agent 协作
- 代码任务委托

### 3）Agent 分工

我没有把所有事情塞给一个 Agent，而是拆成两个核心角色。

#### `main`

主助手负责：

- 对话和需求理解
- 读取工作区记忆文件
- 把自然语言需求整理成结构化任务
- 在需要时委托给其他 Agent
- 汇总结果并回传给我

#### `coding-orchestrator`

这是我自己补的专用 Agent，职责集中在代码任务执行链路：

- 管理仓库注册表
- 自动 clone 和自动注册仓库
- 解析结构化代码任务
- 创建隔离 worktree
- 选择执行器
- 落盘保存 job 输入、日志和结果
- 把结果摘要回传给上层

### 4）渠道接入

我现在主要接两个渠道：

- Telegram
- 钉钉

Telegram 接入标准、上线快。钉钉更重要，因为它本来就在我的日常工作沟通链路里。

在钉钉侧我做了几件体验优化：

- Markdown 输出
- thinking 展示
- 自定义“思考中”提示语
- card 模板支持

### 5）Hook 和 Cron

这两个能力我都打开了。

#### Hook

用于接收外部结构化请求。无论请求来自聊天、脚本还是外部系统，只要能整理成标准 payload，就可以投递到同一条任务链路。

#### Cron

用于定时任务，例如：

- 定期提醒
- 巡检
- 周期性整理
- 定时触发 Agent 执行

打开 Hook 和 Cron 后，OpenClaw 就不再是纯聊天入口，而是一个可被人和系统共同触发的执行运行时。

## 架构怎么拆

我把系统拆成五层：

```text
[聊天入口层]
  ├─ Telegram
  └─ 钉钉

[会话与代理层]
  ├─ main
  └─ coding-orchestrator

[能力工具层]
  ├─ 文件读写
  ├─ shell/runtime
  ├─ browser
  ├─ memory
  ├─ sessions / subagents
  └─ messaging plugins

[任务编排层]
  ├─ hooks
  ├─ cron
  ├─ structured job request
  └─ agent-to-agent delegation

[落盘与工作区层]
  ├─ workspace
  ├─ memory files
  ├─ repo registry
  ├─ jobs/
  └─ worktrees/
```

聊天只是入口，不是系统本体。真正决定系统稳定性的，是这三件事：

- 职责分层
- 任务结构化
- 结果持久化

### 为什么把编码能力从 `main` 拆出去

一开始我也试过让主助手直接改代码。短期能跑，但任务一多就会出现典型问题：

- 直接碰业务仓库，容易把现场弄乱
- 多任务共用目录，改动互相污染
- 任务输入、日志、结果没有统一落点
- 过几天回看时，难定位“这次改动是谁在什么上下文里做的”

所以现在我把边界定得很清楚：

主助手负责“理解与沟通”，编码链路交给专门编排层。

### 任务链路

当前的代码任务流转如下：

```text
聊天渠道收到自然语言需求
        ↓
main 解析需求
        ↓
归一化为结构化代码任务
        ↓
委托给 coding-orchestrator
        ↓
准备 repo / worktree / executor
        ↓
执行并写入 job 结果
        ↓
返回摘要给 main
        ↓
main 再反馈给用户
```

这条链路里最关键的不是“选哪个模型”，而是先把自然语言收敛成稳定任务块。

### 统一代码任务块

我现在尽量把请求压成统一格式：

```text
Coding job request
repoKey: xxx
goal: xxx
constraints: xxx
acceptanceCriteria: xxx
executorPreference: auto
approvalMode: safe-edit
```

这样做的直接收益是：

- 用户换说法，下游也能稳定理解
- Hook 请求和聊天请求可以共用一条执行链路
- 日志、追踪和复盘都有固定字段

### 文件化记忆的价值

OpenClaw 的记忆不是全塞进黑盒数据库，而是落在文件里：

- `MEMORY.md`：长期记忆
- `memory/YYYY-MM-DD.md`：每日记录
- `SOUL.md`：助手行为基调
- `USER.md`：协作信息
- `AGENTS.md` / `TOOLS.md`：规则和环境约束

文件化记忆的好处是透明：能看、能改、能版本管理。

## 我补上的几项核心能力

### 1）新增 `coding-orchestrator`

我的目标不是再造一个“会写代码的 Agent”，而是补齐代码任务的工程化外壳。

主要能力：

- 仓库注册表
- 自动 clone + 自动注册
- 自然语言到结构化任务解析
- 执行器路由
- 每任务独立 worktree
- 输入、日志、结果统一落盘

### 2）建立统一任务协议

如果没有协议，系统很快会变成“聊天一套字段、Hook 一套字段、CLI 一套参数”。短期还能靠记忆维护，长期一定失控。

统一协议的价值在于：

- `main` 做归一化更稳定
- `coding-orchestrator` 接到的是明确字段，不是自由描述
- 新入口（聊天、Hook、面板）都能落到同一个任务模型

### 3）引入 `approvalMode`

很多 AI coding 问题不在“能不能改”，而在“允许改到什么程度”。

我把授权边界拆成三层：

- `plan-only`：只分析，不改代码
- `safe-edit`：默认模式，允许改动但边界保守
- `full-edit`：明确放权，适合已确认的大改

这样执行系统不需要猜用户语气，直接按字段动作。

### 4）双轨落盘：`worktrees/` + `jobs/`

我现在固定用双轨结构：

- `worktrees/<jobId>/`：真实代码副本
- `jobs/<jobId>/`：任务元数据和产物

`jobs/<jobId>/` 里会保存：

- job 描述
- prompt
- 执行日志
- 结果摘要

这样每次任务都有明确归档对象，后续排障和复盘成本会低很多。

### 5）通过 Hook 接入外部触发

有了 Hook，任务来源不再只靠聊天框，后续可以接：

- 本地脚本
- Web 面板
- 外部系统回调
- 工单流或发布流

这一步让 OpenClaw 更像一个事件驱动执行层，而不是聊天插件。

### 6）扩展钉钉接入

钉钉接入对我不是“多一个平台”，而是把 Agent 放进我本来就在用的工作环境。

除了消息收发，我更在意这几个点：

- 输出格式是否可读
- thinking 展示是否清楚
- 提示语是否符合日常习惯
- 卡片模板是否能承载结构化结果

目前这一层已经可用，但后续还要继续打磨消息工具层和插件层的边界。

## 目前看到的主要问题

### 1）安全边界还可以收紧

当前配置偏个人强权限，效率高，但如果扩到多人协作，还需要继续补：

- 群聊策略
- 高权限工具分级
- 文件系统边界控制
- runtime 工具暴露范围
- 插件能力分层

### 2）钉钉插件链路还不够稳

现在能用，但要做成长期入口，还要解决：

- 富消息一致性
- 媒体发送抽象
- 卡片能力的统一接入
- 插件层与通用消息工具层的分工

### 3）`coding-orchestrator` 还有扩展空间

我接下来会优先补这几块：

- 更细的风险分级
- 更丰富的执行器路由策略
- 自动测试/构建挂钩
- 更稳定的结果摘要
- job 查询面板
- 与 Web 管理界面的整合

## 小结

我们现在想要的不是“一个会聊天的 AI”，而是一套能长期协作的本地 Agent 系统。

具体来说，它要能理解自然语言需求，转成结构化任务，调用真实工具执行，并且把过程和结果留在可追踪的位置。

OpenClaw 还不完美，但这套形态已经能稳定支撑我的日常工作。后面继续补安全模型、插件抽象和编排能力，整体就会更扎实。
